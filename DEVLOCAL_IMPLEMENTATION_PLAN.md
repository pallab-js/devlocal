## P0 — Critical Bugs

### P0-001 — `update_tray_menu` IPC call is unregistered (runtime crash)

**File**: `crates/tauri-shell/src/lib.rs`  
**File**: `packages/desktop/src/lib/ipc.ts`

**Problem**: `ipc.ts` exports `updateTrayMenu: (containers: ContainerInfo[]) => invoke<void>("update_tray_menu", { containers })`. This is called from `Dashboard.tsx` on every container list refresh. The Tauri command `update_tray_menu` is **not in `tauri::generate_handler![]`** in `lib.rs`. Calling an unregistered command returns a Tauri error at runtime (the promise rejects), and Dashboard.tsx silently catches it with `.catch(() => {})` — so this fails invisibly every poll cycle.

**Fix**:

Option A (recommended): Implement `update_tray_menu` command in `tauri-shell/src/docker.rs` and register it:

```rust
// crates/tauri-shell/src/docker.rs
#[tauri::command]
pub async fn update_tray_menu(
    containers: Vec<docker_ops::ContainerInfo>,
    app: tauri::AppHandle,
) -> crate::error::Result<()> {
    use tauri::menu::{Menu, MenuItem, PredefinedMenuItem};
    use tauri::tray::TrayIconExt;
    // Rebuild tray menu: top 5 running containers with start/stop shortcuts
    let running: Vec<_> = containers.iter().filter(|c| c.state == "running").take(5).collect();
    let open = MenuItem::with_id(&app, "open", "Open DevOpsLocal", true, None::<&str>)?;
    let sep = PredefinedMenuItem::separator(&app)?;
    let quit = MenuItem::with_id(&app, "quit", "Quit", true, None::<&str>)?;
    let mut items: Vec<&dyn tauri::menu::IsMenuItem<_>> = vec![&open, &sep];
    // ... add container items ...
    let menu = Menu::with_items(&app, &[&open, &sep, &quit])?;
    if let Some(tray) = app.tray_by_id("main") {
        tray.set_menu(Some(menu))?;
    }
    Ok(())
}
```

Then add to `tauri::generate_handler![]` in `lib.rs`:
```rust
docker::update_tray_menu,
```

Option B (quick fix): Remove `updateTrayMenu` from `ipc.ts` and the call in `Dashboard.tsx` until Option A is implemented.

### P0-002 — Dynamic Tailwind class names will be purged in production

**File**: `packages/desktop/src/pages/Dashboard.tsx`

**Problem**: `containerRowActions` function (lines ~40–43 in Dashboard.tsx) builds Tailwind class names by string manipulation:

const containerRowActions = (color: string, bg: string) =>

const ACTION_CLASSES = {

## P1 — Correctness & Security

### P1-001 — `import_secrets` (Kubernetes format) bypasses keychain for sensitive keys

**File**: `crates/db/src/commands.rs` — `import_secrets` function, `"kubernetes"` branch

**Problem**: `upsert_env_var` correctly routes values whose keys match `is_sensitive()` to the OS keychain (stores sentinel `"__keychain__"` in SQLite). However the `import_secrets` Kubernetes branch directly inserts raw decoded values into the DB via a raw `sqlx::query` without calling `is_sensitive()`. A `DB_PASSWORD` or `API_TOKEN` from a Kubernetes Secret is stored in plaintext in SQLite.

**Fix**: After decoding the base64 value, run the same keychain logic as `upsert_env_var`:
```rust
let stored_value = if is_sensitive(&key) {
    let keyring_key = format!("{scope}:{key}");
    let entry = keyring::Entry::new(KEYRING_SERVICE, &keyring_key)
        .map_err(|e| e.to_string())?;
    entry.set_password(&value).map_err(|e| e.to_string())?;
    "__keychain__".to_string()
} else {
    value
};
// then use stored_value in the INSERT
```

### P1-002 — `compose_up` / `compose_down` accept unsanitised filesystem paths

**Files**: `crates/docker-ops/src/lib.rs` (`compose_up`, `compose_down`), `crates/tauri-shell/src/docker.rs`

**Problem**: Both functions accept a `config_dir: &str` from the Tauri command (user-provided) and pass it directly to `tokio::process::Command::new("docker").current_dir(config_dir)`. There is no validation that `config_dir` is within an expected location. An attacker or malicious prompt could supply `/etc` or `~/` to run `docker compose up` in unexpected directories.

**Fix in `tauri-shell/src/docker.rs`** (validate before calling docker-ops):
```rust
#[tauri::command]
pub async fn compose_up(project_name: String, config_dir: String) -> crate::error::Result<()> {
    // Validate config_dir is an existing directory and is absolute
    let path = std::path::Path::new(&config_dir);
    if !path.is_absolute() || !path.is_dir() {
        return Err(crate::error::AppError::Generic(
            "config_dir must be an absolute path to an existing directory".into()
        ));
    }
    // Optionally restrict to user home
    Ok(docker_ops::compose_up(&project_name, &config_dir).await?)
}
```
Apply same fix to `compose_down`.

### P1-003 — Polling intervals depend on `localStorage` pre-population

**File**: `packages/desktop/src/hooks/useQueries.ts`

**Problem**: `useContainers` and `useHostStats` read poll intervals directly from `localStorage`:
```ts
const interval = Number(localStorage.getItem("poll_containers") ?? 5000);
```
On first app launch (or in a fresh profile), `localStorage` is empty. The Settings page loads DB values and syncs them to `localStorage` on mount — but only if the Settings page has been visited. If the user never visits Settings, `localStorage` never gets set, and the app silently uses hardcoded defaults (5000ms / 3000ms). This is fine as a fallback, but the issue is the pattern: if a user changes their intervals in DB then clears localStorage (browser dev tools, etc.), their settings are lost.

**Fix**: In the Settings page's `useEffect`, always write DB values to localStorage on mount (already done partially). Additionally, expose a React Context or Zustand slice that holds `{ pollContainers, pollStats }` and is initialised from DB at app startup in `main.tsx`, so hooks consume that rather than reading localStorage directly.

Minimal fix (add to `main.tsx` startup):
```ts
// main.tsx — after React root mounts, initialise poll settings
Promise.all([ipc.getSetting("poll_containers"), ipc.getSetting("poll_stats")]).then(([c, s]) => {
  if (c) localStorage.setItem("poll_containers", c);
  if (s) localStorage.setItem("poll_stats", s);
}).catch(() => {});
```

### P1-004 — `ipc.ts` type definitions can diverge from `ts-rs` generated types

**Files**: `packages/desktop/src/lib/ipc.ts`, `packages/shared/types/*.ts`

**Problem**: `ipc.ts` manually re-declares all TypeScript interfaces (`ContainerInfo`, `HostStats`, etc.). The `ts-rs` crate auto-generates the same types into `packages/shared/types/`. If a Rust struct field changes (e.g., a new field added to `ContainerInfo`), the generated type in `packages/shared/types/ContainerInfo.ts` updates but `ipc.ts` does not, causing silent runtime deserialization mismatches.

**Fix**: Import types from the shared package in `ipc.ts` instead of re-declaring them:
```ts
// packages/desktop/src/lib/ipc.ts
export type { ContainerInfo } from "@devopslocal/shared/types/ContainerInfo";
export type { ContainerDetails } from "@devopslocal/shared/types/ContainerDetails";
// ... etc
```
Ensure `packages/shared/package.json` exports the types correctly. Add a CI step: `cargo test --workspace` (which runs `ts-rs` export) then `git diff --exit-code packages/shared/types/` to catch drift.

### P2-001 — Extract Docker state helper to eliminate 20+ duplicate guard blocks

**File**: `crates/tauri-shell/src/docker.rs`

**Problem**: Every command begins with:
```rust
let docker = docker.0.as_ref()
    .ok_or_else(|| crate::error::AppError::Generic("Docker not connected".to_string()))?;
```
This pattern appears 15+ times. It is verbose, error-prone, and produces inconsistent error messages.

**Fix**: Add a private helper at the top of `docker.rs`:
```rust
fn require_docker(state: &DockerState) -> crate::error::Result<Arc<Docker>> {
    state.0.as_ref()
        .map(Arc::clone)
        .ok_or_else(|| crate::error::AppError::Generic("Docker daemon not connected".into()))
}
```
Replace all instances:
```rust
// Before
let docker = docker.0.as_ref().ok_or_else(|| ...)?;
// After
let docker = require_docker(&docker)?;
```

### P2-003 — Cache host disk stats separately from CPU/RAM

**File**: `crates/docker-ops/src/lib.rs` — `get_host_stats`  
**File**: `crates/tauri-shell/src/docker.rs` — `get_host_stats` command

**Problem**: `get_host_stats` calls `Disks::new_with_refreshed_list()` on every invocation. Disk usage changes rarely (every few seconds). Rebuilding the disk list on a 3-second poll is wasteful, especially with many mount points.

**Fix**: Add a second managed state `DiskState(Mutex<(Disks, Instant)>)` in `lib.rs`. In `get_host_stats`, only refresh disks if more than 10 seconds have elapsed since last refresh:
```rust
pub struct DiskState(pub Mutex<(sysinfo::Disks, std::time::Instant)>);

// In command:
let (disk_used, disk_total) = {
    let mut guard = disk_state.0.lock()?;
    if guard.1.elapsed() > std::time::Duration::from_secs(10) {
        guard.0.refresh();
        guard.1 = std::time::Instant::now();
    }
    guard.0.iter().fold((0u64, 0u64), |(u, t), d| {
        (u + (d.total_space() - d.available_space()), t + d.total_space())
    })
};
```

### P3-006 — Structured IPC error types

**Files**: `crates/tauri-shell/src/error.rs`, `packages/desktop/src/lib/ipc.ts`

**Problem**: All Tauri commands return `Result<T, String>`. The frontend catches generic strings like `"Docker daemon not connected"` or `"No such container: abc123"`. There is no way to programmatically distinguish a "Docker unavailable" error from a "container not found" error, so the UI can only show a raw string.

**Fix**:

In `error.rs`, `AppError` already exists — ensure it serialises with a `code` field:
```rust
#[derive(Debug, Serialize, thiserror::Error, TS)]
#[serde(tag = "code", content = "message")]
pub enum AppError {
    #[error("Docker not connected")]
    DockerNotConnected,
    #[error("Container not found: {0}")]
    NotFound(String),
    #[error("Docker error: {0}")]
    Docker(String),
    #[error("{0}")]
    Generic(String),
}
```
In `ipc.ts`, add an `AppError` type and a typed error handler:
```ts
export interface AppError { code: string; message: string; }
export function isAppError(e: unknown): e is AppError {
  return typeof e === "object" && e !== null && "code" in e;
}
```
Pages can then show contextual messages: `isAppError(e) && e.code === "DockerNotConnected" ? <DockerOfflineBanner /> : <GenericError />`.

### P4-005 — Add ts-rs drift detection to CI

**File**: `.github/workflows/ci.yml`

**Problem**: If a Rust struct changes, `ts-rs` must be re-run (`cargo test`) to regenerate `packages/shared/types/*.ts`. If this is forgotten, `ipc.ts` or pages importing from shared will use stale types. There is no CI guard.

## Key invariants (never violate)

1. **All SQL uses parameterised binds** — no string interpolation in queries. `sqlx::query` only.
2. **Sensitive env var values go through keychain** — any code path that writes an env var must call `is_sensitive()` and branch accordingly.
3. **No `unwrap()` in Tauri commands** — use `?` with the `AppError` type.
4. **Tailwind classes must be static strings** — no dynamic class construction via string interpolation.
5. **ts-rs types are source of truth** — `ipc.ts` imports from `@devopslocal/shared`, not re-declares.
6. **All Tauri commands must be in `generate_handler![]`** — verify after every new command.
7. **Docker availability is optional** — app must start and show a graceful error if Docker daemon is not running (already implemented; don't break this).

## Context for AI: how this codebase works

- Rust types annotated with `#[derive(TS)]` auto-export to `packages/shared/types/*.ts` on `cargo test`.
- `ipc.ts` is the **only** place the frontend calls Rust. All `invoke<T>("command_name", args)` calls live here.
- CSS design tokens live in `src/styles/tokens.css` as a Tailwind v4 `@theme {}` block. Colors like `text-green`, `bg-orange-dim`, `border-error-dim` are custom tokens, not standard Tailwind palette.
- The `SysState` mutex wraps `sysinfo::System`. It uses `std::sync::Mutex` — do NOT hold the guard across an `.await` point.
- Docker state (`DockerState`) wraps `Option<Arc<bollard::Docker>>`. `None` means daemon unavailable. All commands must handle `None` gracefully.
- Log streaming: backend spawns a tokio task per container, stored in `LogStreamState (Mutex<HashMap<containerId, JoinHandle>>)`. `stop_logs` aborts the task.
- Event streaming: one task for Docker daemon events, stored in `EventStreamState(Mutex<Option<JoinHandle>>)`.
- DB pool: `OnceLock<SqlitePool>` global in `crates/db/src/lib.rs`. WAL mode enabled. 5 max connections.
- Migrations: `crates/db/migrations/` numbered SQL files. SQLx runs them on startup via `sqlx::migrate!()`.
